# 项目全局规则

## 1. 概述

本文档定义了项目的全局开发规则，所有参与项目开发的人员必须遵守。这些规则旨在确保代码的一致性、可维护性和可扩展性，同时提高开发效率。

## 2. API文档优先原则

### 2.1 核心原则

**后续代码开发优先读取API文档文件夹中的内容进行开发，避免反复读取源码。**

### 2.2 具体要求

1. **开发前必读**：在开始任何新功能开发或修改现有功能之前，必须先阅读`api-docs/`目录下的相关文档，了解系统架构、模块功能和API接口。

2. **文档优先**：每次生成代码前，都必须通过文档来了解项目，尽量不要通过阅读代码来理解项目结构和功能。

3. **文档为准**：API文档是项目的权威参考，当文档描述与实际代码存在差异时，应优先按照文档描述进行开发，并及时更新代码以匹配文档。

4. **文档同步更新**：每次生成代码后，必须增加或修改相关文档，确保文档与代码保持一致。任何对代码的修改，如添加新功能、修改现有功能或调整API接口，都必须同步更新对应的API文档。

5. **避免直接依赖源码**：开发新模块时，应基于API文档中定义的接口进行设计和实现，避免直接依赖其他模块的内部实现细节。

## 3. 代码开发流程

### 3.1 新功能开发流程

1. 阅读相关API文档，了解系统架构和现有模块
2. 设计新功能的API接口，更新或创建相关文档
3. 实现新功能，遵循代码规范
4. 测试新功能，确保符合API文档描述
5. 提交代码和文档更新

### 3.2 现有功能修改流程

1. 阅读相关API文档，了解现有功能的设计和实现
2. 分析修改需求，确定是否需要修改API接口
3. 如果需要修改API接口，先更新API文档
4. 实现代码修改
5. 测试修改后的功能，确保符合API文档描述
6. 提交代码和文档更新

## 4. 命名规范

### 4.1 文件命名

- 采用小写字母和连字符(-)分隔，例如：`keyboard-controller.js`
- 类文件使用大驼峰命名法，例如：`Player.js`
- 配置文件使用小写字母，例如：`config.js`

### 4.2 类和函数命名

- 类名采用大驼峰命名法，例如：`KeyboardController`
- 函数名和方法名采用小驼峰命名法，例如：`controlPlayer`
- 常量名采用全大写和下划线分隔，例如：`CANVAS_WIDTH`

### 4.3 变量命名

- 变量名采用小驼峰命名法，例如：`gameState`
- 布尔值变量名以动词开头，例如：`isRunning`, `hasCollision`

## 5. 代码结构规范

### 5.1 模块划分

- 按功能模块划分文件，每个文件只包含一个主要功能或类
- 相关模块放在同一个子目录下，例如：`classes/`目录包含所有游戏对象类
- 避免创建过大的文件，单个文件代码行数不应超过200行

### 5.2 代码组织

- 每个文件顶部包含模块描述和依赖导入
- 类定义按"属性声明 → 构造函数 → 方法"的顺序组织
- 方法按功能相关性分组
- 代码注释清晰，解释复杂逻辑和关键算法

### 5.3 依赖管理

- 使用ES6模块系统(`import`/`export`)管理依赖
- 避免循环依赖
- 只导入需要的功能，例如：`import { Player } from './classes/Player.js'`

## 6. 代码质量要求

### 6.1 可读性

- 代码结构清晰，逻辑分明
- 使用适当的缩进和空行，提高代码可读性
- 变量和函数命名具有描述性，避免使用缩写或模糊的命名

### 6.2 可维护性

- 代码模块化，便于修改和扩展
- 避免硬编码，使用配置常量
- 分离业务逻辑和视图渲染

### 6.3 性能

- 优化游戏循环，避免不必要的计算
- 使用高效的算法和数据结构
- 避免内存泄漏，及时清理不再使用的对象

## 7. 测试要求

- 新功能必须经过测试，确保符合预期行为
- 测试用例应覆盖正常情况和边缘情况
- 测试结果应与API文档描述一致

## 8. 文档管理

- API文档应简洁明了，重点描述接口和功能
- 文档结构清晰，便于查找和阅读
- 使用Markdown格式编写文档
- 文档应包含示例代码，便于理解和使用

## 9. 版本控制

- 遵循Git版本控制规范
- 提交信息清晰，描述修改内容
- 定期合并分支，保持代码库的整洁

## 10. 违规处理

违反上述规则的代码将被要求修改，直到符合要求为止。严重或多次违反规则的开发人员将被限制参与项目开发。

## 11. 规则更新

本规则将根据项目发展和实际需求进行定期更新，所有更新将通过团队会议或邮件通知所有开发人员。
